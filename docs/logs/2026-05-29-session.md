## What was done

### 1. Auto-Tune for the community simulator (committed)

Added a static, model-free parameter search to `community-sim/`. Commit
`47dbf96` "Add browser-side Auto-Tune to community simulator".

- **UI** (`index.html`, `styles.css`): an Auto-Tune panel with a fixed
  "Healthy Balance" target, a Quick/Normal/Deep search budget, a Run tune
  button, status text, and ranked candidate cards (score + metrics +
  explanation + Apply).
- **Search** (`app.js`): a deterministic randomized/grid hybrid
  (`makeCandidates`) — current settings + balanced anchors over the commit-rule
  × exchange-rate options + randomized fill across conservative ranges. The 13
  tunable controls only; population and run speed held fixed so candidates stay
  comparable.
- **Off-screen evaluation** (`evaluateRaw`): saves the live `state`, seeds a
  candidate state, runs a fixed horizon, averages recent-cycle metrics, then
  restores. The visible sim is never mutated until a candidate is applied.
- **Scoring** (`scoreMetrics`): the "Healthy Balance" objective is a weighted
  sum of five normalized components — deal trust (0.30), satisfaction (0.26),
  demand coverage (0.18), moderate-inequality "Gini health" peaking at Gini≈0.30
  (0.16), and admission activity (0.10).
- **Refactors enabling headless runs**: `params()` split into
  `rawFromControls()` + `normalizeParams()`; `reset()` split into `seedState()`
  + `render()`; `step()` factored into `advanceCycle(p)`; `allocateResources()`
  now takes an optional `p`. Live behavior unchanged.

**Verification**: `node --check` passes. A DOM-stub Node harness (`vm` context)
confirmed: high objection pressure scores lower than low (0.518 vs 0.545, admits
10/10 → 5/30); both fixed and floating exchange modes evaluate finitely; 14
candidates all finite with differentiated scores (top 0.733 vs base 0.545);
re-runs deterministic; live `state.cycle` stays 0 after tuning (sim untouched).
No in-browser smoke test — no browser/puppeteer available in the VM.

### 2. Visualization discussion (deferred)

Investigated "what is everyone doing?" Found the **network edges are fake** —
`drawNetwork` uses `randForPair` (a per-cycle ID hash), so they show no real
relationship. Noted the real activity data already exists in
`state.attestations` (with cycle + participant IDs) and that the goods market is
**pooled, not bilateral** (no per-seller trade edges exist).

Recommended a split: superimpose *real* social flows (endorsement/objection/
recognition) on the existing network, and add a *separate* economic panel
(bipartite stewards↔goods flow and/or a per-steward ledger), since pooled trade
does not map to point-to-point lines. User agreed on direction but chose to
**delay implementation**.
